You are an elite software engineer and code auditor reviewing this entire project as if it were a production financial application handling real money.

Read **every file** completely before making conclusions. Trace execution across files, APIs, database operations, frontend, backend, scheduled jobs, authentication, and business logic. Do not review files in isolation.

Your primary objective is **not** to find syntax errors or style issues. Your objective is to uncover the **hardest-to-spot logic bugs** that could survive testing and appear only in production.

Think through every possible execution path and actively try to break the application.

Look for:

* Hidden business logic flaws
* Race conditions and concurrency issues
* Incorrect state transitions
* Multi-tab and duplicate request problems
* Double processing (deposits, withdrawals, purchases, rewards, referrals, etc.)
* Edge cases caused by refreshes, retries, crashes, or network failures
* Missing validation
* Authorization and privilege logic mistakes
* Database consistency problems
* Transaction and rollback issues
* Time, timezone, and scheduling bugs
* Frontend/backend state mismatches
* Cross-file assumptions that can fail
* Features that work individually but break when combined
* Any logic that could cause users to gain, lose, duplicate, or corrupt data

Do not stop after finding obvious issues. Continue searching until you have thoroughly reviewed the entire project.

Then, identify anything that isn't technically a bug but could be redesigned to make the application more robust, scalable, secure, maintainable, performant, or production-ready.

For every finding, provide:

* Severity (Critical, High, Medium, Low, or Improvement)
* File(s) and function(s)
* Why it's a problem
* A realistic scenario where it fails
* The recommended production-grade fix
* Why your fix is the best approach

Do not invent issues. Every finding must be supported by evidence from the code. If you're uncertain, say so instead of guessing.

Your goal is to find the kinds of bugs that experienced engineers commonly miss during normal reviews.
